Chapter 1 of 7

Why dbt?

dbt matters because analytics SQL eventually stops being “just queries” and starts becoming a system of interconnected transformations.

Standardization

One shared language

dbt creates a unified workflow for organizing transformations, testing assumptions, generating documentation, and collaborating across warehouses and teams.

Abstraction

Focus on the transformation

Models are usually declarative SELECT statements while dbt handles materialization and orchestration boilerplate behind the scenes.

Layered modeling

Stable transformation layers

Staging, intermediate, and mart layers create stable boundaries between transformations and isolate upstream changes from downstream consumers.

Lineage

See how everything connects

dbt automatically tracks upstream dependencies and downstream consumers, turning scattered SQL into a visible transformation graph.

Dependencies & orchestration

Correct build order

dbt determines what needs to run first and coordinates execution in the warehouse, removing fragile manual ordering.

Testing & documentation

Contracts beside the logic

Assertions and documentation live alongside the transformations themselves instead of drifting into wikis, spreadsheets, or tribal knowledge.

One-line takeaway: dbt turns SQL transformations into a standardized, layered, dependency-aware system.

Install

Getting from nothing to a working project skeleton.

Installation

dbt has a core package and adapter-specific packages. You install the adapter for your warehouse — the adapter pulls in dbt-core automatically.

# For DuckDB (great for local dev) pip install dbt-duckdb # For Snowflake, BigQuery, Redshift, Postgres... pip install dbt-snowflake pip install dbt-bigquery pip install dbt-postgres

Creating a project

dbt init my_project # Follow the prompts: choose your adapter, name your project

This generates a project skeleton.

Connection config
~/.dbt/ └── profiles.yml
Project repo
my_project/ ├── dbt_project.yml ├── README.md ├── analyses/ ├── macros/ ├── models/ ├── seeds/ ├── snapshots/ └── tests/
One-line takeaway: dbt init creates the skeleton for a structured, shareable transformation project.
Chapter 3 of 7

Project vs profiles

This is the most important conceptual distinction in dbt. Get this and everything else clicks.

dbt_project.yml

What to build

Lives inside your repo. Contains project name, model configs, variable defaults, and resource settings. Gets committed to Git and shared with your team.

profiles.yml

How to connect

Lives in ~/.dbt/ on your machine. Contains credentials, host, database, schema, and environment targets. Never committed. Different per person/environment.

Why this split? SQL transformation logic is shareable. Credentials are not. dbt keeps them physically separate so you can push code to GitHub without leaking passwords. In CI/CD, credentials usually come from environment variables injected at runtime.

What dbt_project.yml actually looks like

# Name your project! Project names should contain only lowercase characters # and underscores. A good package name should reflect your organization's # name or the intended use of these models name: 'dbt_test_project' version: '1.0.0' # This setting configures which profile dbt uses for this project. profile: 'dbt_test_project' # These configurations specify where dbt should look for different types of files. # The `model-paths` config, for example, states that models in this project can be # found in the "models/" directory. You probably won't need to change these! model-paths: ["models"] analysis-paths: ["analyses"] test-paths: ["tests"] seed-paths: ["seeds"] macro-paths: ["macros"] snapshot-paths: ["snapshots"] clean-targets: # directories to be removed by `dbt clean` - "target" - "dbt_packages" # Configuring models # Full documentation: https://docs.getdbt.com/docs/configuring-models # In this example config, we tell dbt to build all models in the example/ # directory as views. These settings can be overridden in the individual model # files using the `{{ config(...) }}` macro. models: dbt_test_project: # Config indicated by + and applies to all files under models/example/ example: +materialized: view

What profiles.yml looks like

dbt_test_project: outputs: dev: type: duckdb path: dev.duckdb threads: 1 prod: type: duckdb path: prod.duckdb threads: 4 target: dev
One-line takeaway: dbt_project.yml defines the transformation project; profiles.yml defines how dbt connects to the warehouse.

Chapter 5 of 7

Model path

Models are the heart of dbt. Most projects follow a layered path from sources → staging → intermediate → marts.

Sources
Raw data. Use source() when pointing at warehouse tables dbt did not create.
↓
Staging models
Clean & rename. Usually built from source(). One staging model per source table. Just clean and rename. No joins in staging.
↓
Intermediate models
Join & enrich. Usually built from upstream models via ref().
↓
Marts
Business logic. Final business-facing models.
↓
BI Tools / Consumers
Analysts query.
One-line takeaway: dbt projects organize transformations into stable layers between raw data and business-facing outputs.
Chapter 6 of 7

Materializations

A model starts as a SELECT statement. Materialization tells dbt how to persist it in the warehouse.

view

Logical query

Cheap to build. Recomputed when queried. Good for staging and lightweight transforms.

table

Physical result

Stored as a table. Faster for consumers, but must be rebuilt.

incremental

Append/update new data

Only processes new or changed records after the first run. Important for large datasets.

ephemeral

Inlined CTE

Not stored separately. dbt injects the SQL into downstream models.

Materializations are often configured by layer

# dbt_project.yml models: taxi_rides_ny: staging: +materialized: view marts: +materialized: table
One-line takeaway: Materialization controls how dbt persists a model inside the warehouse.
Chapter 7 of 7

Commands

Commands make more sense once you know what dbt is operating on.

Core commands

dbt debug # Always run this first. Validates your profile connection and project config. dbt run # Executes all models. Builds tables/views in your warehouse. dbt run --select stg_orders # Run just one model dbt run --select staging.* # Run all models in the staging/ folder dbt run --select +mart_revenue # Run mart_revenue AND all its upstream dependencies dbt test # Runs all tests. Fails the command if any test returns rows. dbt build # run + test + seed + snapshot in dependency order. The "do everything" command. dbt compile # Renders Jinja but doesn't run anything. Great for debugging SQL. dbt seed # Loads CSV files from seeds/ into the warehouse. dbt snapshot # Runs snapshot logic. dbt docs generate && dbt docs serve # Generates the data catalog and serves it locally at localhost:8080 dbt deps # Installs packages from packages.yml (like npm install). dbt source freshness # Checks that source tables have been updated recently.

The selector syntax (--select)

The --select flag is incredibly powerful. Selectors can be combined with spaces (union) or commas.

# Downstream: model + everything that depends on it dbt run --select mart_orders+ # Upstream: everything mart_orders depends on dbt run --select +mart_orders # Between two nodes dbt run --select +mart_orders+2 # (up to 2 layers downstream) # By tag dbt run --select tag:nightly # Modified models only (great in CI) dbt run --select state:modified+
The through line dbt commands operate on the project graph. They are not random CLI verbs; they are actions against models, sources, seeds, snapshots, tests, docs, and the dependency graph created by ref() and source().
One-line takeaway: dbt commands operate on the transformation graph defined by your project.
